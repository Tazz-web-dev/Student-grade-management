1  Introduction
1.1  Purpose
This document defines the standardized specification for the Software Development Document that every student must submit as part of the Software Engineering course assignment. It establishes the structure, formatting rules, content requirements, and quality standards for the deliverable, drawing from established graduation thesis (毕业设计论文) conventions while adapting them to the specific needs of this course.
The specification is designed to ensure that all submitted documents follow a consistent, professional format that demonstrates systematic software engineering practices across the entire development lifecycle.
1.2  Scope
This specification applies to all students enrolled in the Software Engineering course. The guidelines cover:
Document structure and organization
Formatting and layout requirements (fonts, margins, spacing, pagination)
Content requirements for each mandatory section
Rules for figures, tables, code listings, and references
Submission and quality assurance requirements

2.4  Page Numbering
Page numbering follows these conventions:
Front matter (TOC, abstract): Lowercase Roman numerals (i, ii, iii...), centered at page bottom.
Main body (Chapter 1 onward): Arabic numerals (1, 2, 3...), centered at page bottom.
The cover / title page must NOT display a page number.
Page number font: Times New Roman, 10 pt.
2.5  Heading Numbering Scheme
All headings use a hierarchical decimal numbering scheme:
1  Introduction                       (first-level, Heading 1)    1.1  Purpose                      (second-level, Heading 2)        1.1.1  Background Context      (third-level, Heading 3)2  Requirements Analysis               (first-level, Heading 1)    2.1  Functional Requirements        (second-level, Heading 2)...
Do not place a period after the heading number. Do not skip heading levels.
2.6  Figures and Tables
Figures and tables must be formatted as follows:
Figure captions go BELOW the figure. Format: "Figure 2.1  System Architecture Diagram"
Table captions go ABOVE the table. Format: "Table 3.1  User Table Schema"
Numbering is chapter-based: the first figure in Chapter 2 is "Figure 2.1".
All figures and tables must be referenced in the body text.
Insert high-resolution images (minimum 150 DPI). Preferred formats: PNG, SVG.

4  Requirements Analysis Section
This section presents a complete analysis of the software requirements for the selected project. It establishes what the system must do before any design or implementation begins.
4.1  System Requirements Specification
4.1.1  Project Overview
Provide a concise description of the project, its context, and overall goals. Identify the target users and key stakeholders.
4.1.2  Functional Requirements
List all functional requirements using a consistent numbering scheme (e.g., FR-01, FR-02...). Each requirement must be:
Uniquely identified with a requirement ID
Clearly stated in a single, unambiguous sentence
Traceable to use cases in section 4.2
Example functional requirements table:
ID
Name
Description
FR-01
User Registration
The system shall allow new users to register with email and password.
FR-02
User Authentication
The system shall authenticate users with email/password credentials.
FR-03
Book Search
The system shall allow users to search books by title, author, or ISBN.
4.1.3  Non-Functional Requirements
Specify constraints and quality attributes including:
Performance — expected response time, concurrent user capacity
Security — authentication mechanism, data encryption, access control
Usability — target user skill level, interface language
Reliability — uptime expectations, data backup requirements
Technical Constraints — programming language, framework, database, platform
4.2  Use Case Model
4.2.1  Use Case Diagram
Include a UML use case diagram showing all actors and use cases. The diagram must:
Identify all actor roles (e.g., Admin, Librarian, Member)
Show all major use cases as ovals connected to actors
Use proper UML notation (stick figures for actors, solid lines for communication)
Use include / extend / generalization relationships where applicable
Insert the diagram as a high-resolution image (PNG or SVG recommended).

5  Software Design Section
This section translates the requirements into a concrete system design, covering architecture, classes, and data persistence.
5.1  System Architecture Design
Present the high-level architecture:
An architecture diagram showing major components and their interactions
Architectural pattern used (e.g., MVC, Layered Architecture, Client-Server)
Module decomposition and responsibilities
Technology stack overview
Example: Web application architecture using a three-layer model:
Presentation Layer: HTML/CSS/JavaScript frontend (React or Vue.js optional)
Business Logic Layer: REST API service (Python/Flask or Java/Spring Boot)
Data Access Layer: Database (MySQL or PostgreSQL) with ORM

6  Software Implementation Section
6.1  Code Repository
Provide the URL of the GitHub repository containing the complete project source code. The repository must:
Include a README.md with project description, setup instructions, and usage guide
Have meaningful commit messages following a consistent style
Be organized with a clear directory structure
Recommended directory structure for a web application:
project-root/  +-- README.md  +-- requirements.txt / pom.xml / package.json  +-- src/  |   +-- main/  |   |   +-- java/python/...     # application source code  |   |   +-- resources/          # configuration files  |   |   +-- webapp/             # static assets (HTML, CSS, JS)  |   +-- test/                   # unit and integration tests  +-- docs/  |   +-- diagrams/               # UML diagrams, architecture images  |   +-- api-spec/               # API documentation  +-- .gitignore
6.2  Implementation Details
Describe the implementation of the system:
Programming languages, frameworks, and libraries used
How core features were implemented (reference specific classes/modules)
Design patterns applied in the code
Challenges encountered and solutions adopted
Guidelines for code snippets in the document:
Include small code snippets (up to approximately 20 lines) to illustrate key patterns.
Do NOT paste large blocks of source code into the document.
Reference the GitHub repository for full source code access.

8  References and Bibliography
All sources cited in the document must be listed in the References section. Follow the GB/T 7714 standard (Chinese national standard for bibliography), or alternatively use APA or IEEE style consistently throughout the document.
Reference format examples:
Books:
[1] Pressman R S. Software Engineering: A Practitioner's Approach. 8th ed. New York: McGraw-Hill Education, 2014.
Journal Articles:
[2] Sommerville I. Software Engineering. 10th ed. Boston: Pearson, 2016.
Conference Papers:
[3] Beck K. Embracing change with extreme programming. IEEE Computer, 1999, 32(10): 70-77.
Online Resources:
[4] GitHub. Mastering Markdown. https://guides.github.com/features/mastering-markdown/, accessed 2026-05-27.
Reference numbering rules:
Number references sequentially in the order they first appear.
Each in-text citation must have a corresponding entry in the reference list.
Use square brackets for citations: [1], [2-4].
